# 資料與版本更新流程

更新日期：2026-07-21

噴前查有**三層**，各層的更新方式完全不同。搞錯層級是最常見的浪費時間來源。

| 層級 | 內容 | 更新方式 | 使用者何時看到 |
|---|---|---|---|
| ① 資料 | 農藥登記清單（`index.html` 內的 `DATA`） | 改 `index.html` → PR → 合併 | 下次開啟 App |
| ② 網站程式 | 畫面、功能、文案、各 `.js` | 改檔 → PR → 合併 | 下次開啟 App |
| ③ Android 外殼 | TWA 套件（.aab） | 重新產生 → Play Console 送審 | 使用者更新 App |

**重點：①②不用經過 Google 審查。** TWA 只是一層殼，App 打開時載入的是 `searchbefore.tw` 的即時內容，所以資料更新、加功能、修文案全部走 GitHub，合併完就生效。③ 幾乎用不到，見最後一節。

## 一、農藥資料更新（最常做的）

### 資料在哪裡

| 項目 | 位置 |
|---|---|
| 資料本體 | `index.html` 第 985 行 `const DATA={...}`（單行，約 3.6 MB） |
| 資料版本標記 | `index.html` 第 990 行 `const DATA_VERSION="2026-07"` |

### `DATA` 的結構

```
DATA[作物名][病蟲害名] = [ { name, form, content, dilution, phi, dose, times, moa, note, bl }, ... ]
```

規模：**284 種作物 / 2,801 組作物×病蟲害 / 16,342 筆藥劑**。
`bl` 是商品名陣列，`phi` 是安全採收期天數（可為 `null`），`moa` 是 IRAC／FRAC 代碼。

### ✅ `DATA` 的來源（2026-07-21 查明並驗證）

**唯一可用的端點**

```
https://data.moa.gov.tw/Service/OpenData/FromM/PesticideData.aspx?$top=5000&$skip=0
```

支援 `$top` / `$skip` 分頁，可取得全部 **11,869 張**許可證。

| 回傳欄位 | 對應 `DATA` |
|---|---|
| `中文名稱` | `name`（成分中文名） |
| `廠牌名稱` | `bl`（商品名，同用法的多張許可證聚合成陣列） |
| `劑型` | `form` |
| `含量` | `content` |
| `IRAC殺蟲劑抗藥性` / `FRAC殺菌劑抗藥性` / `HRAC除草劑抗藥性` | `moa` |
| `撤銷類別` / `撤銷日期` | 篩掉已廢止的許可證 |
| `農藥使用範圍` | 指向明細端點的網址 |

**使用範圍明細**（每張許可證一次請求）

```
https://data.moa.gov.tw/Service/OpenData/FromM/PesticideDetail.aspx?ltyp={型別}&lno={許可證號}
```

| 回傳欄位 | 對應 `DATA` |
|---|---|
| `作物名稱` | 第一層 key |
| `病蟲害名稱` | 第二層 key |
| `稀釋倍數` | `dilution` |
| `每公頃使用用藥量` | `dose` |
| `施用次數` | `times` |
| `安全採收期` | `phi`（`"18日"` → `18`；`"-日"` → `null`） |
| `備註` | `note` |

### ⛔ 不要用 `api/v1` 那支

```
https://data.moa.gov.tw/api/v1/PesticideDataQueryType/     ← 不可用
```

欄位內容完全相同（英文欄名版），但**非會員只能取得第一頁 500 筆**。第二頁起回傳：

```json
{"RS":"ERROR","MSG":"非會員只限回傳第一頁資料"}
```

`$top` / `$skip` / `offset` / `page` 等參數一律被忽略或拒絕，**無法繞過**。曾誤用這支導致抓取結果為 0，特此記錄。

### 兩個會靜默出錯的坑

**① `撤銷日期` 的空值不是空字串**

未撤銷時回傳的是 `"   /  /  "`（只有空白與斜線的佔位字串）。用 `trim()` 後判斷非空，會把**全部**許可證都當成已撤銷，產出空的 `DATA`。要以「是否含數字」判斷。

**② `安全採收期` 的 `null` 與 `0` 意義完全不同**

`null` 是「未訂或不適用」，`0` 是「當天即可採收」。混為一談會讓倒數功能給出錯誤的可採日期。

### 來源如何驗證的

許可證 `農藥製 00001`「快得保淨」的明細回傳「木瓜／白粉病、稀釋 600 倍、安全採收期 18 日、施用次數 2-3、備註『本試驗加展著劑CS-7，3000倍。』」，與現有 `DATA["木瓜"]["白粉病"]` 中該筆**逐欄一致**（含備註標點）。

### 更新資料的完整流程

```bash
node scripts/fetch-pesticide-source.mjs    # 抓取,可中斷續跑
node scripts/build-data.mjs                # 產生 DATA.built.json(不寫入)
node scripts/compare-data.mjs              # 比對新舊差異 ← 必看
node scripts/build-data.mjs --write        # 確認差異合理後才寫回
npm run release:bump -- <版本> <代號>
node tests/run-all.js
```

`compare-data.mjs` 會特別標出**安全採收期**與**稀釋倍數**的差異——這兩項錯了會直接影響農友的用藥與採收判斷，且不會有任何錯誤訊息。

⚠️ 抓取階段並行數壓在 6、每次請求間隔 60ms。這是政府單位的公開服務，不要調高。


### ✅ 9,999 筆上限（2026-07-21 已修）

農業部端點單頁上限 9,999 筆，直接抓會**靜默少資料**——不報錯，只是回傳剛好 9,999 筆。

已在 `mrl-data/fetch-reference-data.mjs` 加入 `fetchAllPaged()`，以 `$top` / `$skip` 逐頁抓取，並在結果恰為 9,999 筆時直接拋錯（那代表分頁又失效了）。

修正後實測：**11,869 筆**，比原本多 1,870 筆。MRL 對照中部分「查無資料」的判定需以新快照重跑。

## 二、換版（用腳本，不要手改）

```bash
npm run release:bump -- 0.1.12.0 你的用途代號
# 或
node scripts/bump-version.mjs 0.1.12.0 你的用途代號
```

一次改完**四個地方**：

| 檔案 | 內容 |
|---|---|
| `index.html` | `APP_VERSION` |
| `sw.js` | `CACHE_VERSION`（自動附上今天日期） |
| `tests/index-syntax.test.js` | 釘住的 `APP_VERSION` |
| `tests/index-syntax.test.js` | 釘住的快取版本前綴 |

### 為什麼一定要用腳本

版本號散在四處，其中測試檔有**兩個**釘住值。釘 `CACHE_VERSION` 的那一行**整行沒有出現 `CACHE_VERSION` 這個字**（只寫版本字串本身），用關鍵字搜尋根本找不到——這已經害人誤判過一次「這個值沒有測試保護」。

腳本會在任何一處對不上時**整批中止**，不會留下改一半的狀態；也會擋下「版本號沒往上加」。

### 漏改的後果

漏改 `sw.js` 的 `CACHE_VERSION` 而測試釘住值也一起錯的話，使用者手機會**一直吃舊快取**：你在電腦上看到更新了，農友那邊沒有，而且**完全不會報錯**。這是這個專案最難察覺的失敗模式。


## 三、測試與發布

```bash
node tests/run-all.js          # 全部測試，必過才能合併
npm run release:check          # 測試 + 建置 + 發布成品檢查
```

`release:check` 會額外驗證：dist 不含文件或 source map、沒有私鑰外洩、沒有殘留舊網址、`index.html` 引用的 script 與 Service Worker 快取清單都存在。

### GitHub 流程（**永遠不要直接改 `main`**）

1. 從 `main` 開工作分支
2. 改動 + 測試通過
3. Commit，建立 Pull Request
4. 確認差異後合併到 `main`
5. GitHub Pages 自動部署，網站即更新
6. 確認網站正常後刪除分支

### 部署後自我檢查

- 手機開啟網站，確認新資料／新功能出現
- 關網路重開，確認離線仍可用（Service Worker 有正確更新）
- 從 Android App 開啟，確認**上方沒有跑出瀏覽器網址列**（有的話代表 TWA 網域驗證出問題，見下節）

## 四、Android 外殼更新（很少做）

### 什麼時候才需要重新產生 .aab

只有這幾種：

- App 名稱或圖示改了
- Google 每年強制拉高的 **target SDK 門檻**（約每年 8 月底，Play Console 會提前寄信，目前是 35）
- TWA 設定變動（啟動網址、螢幕方向、狀態列顏色）
- 未來要做內購

**資料更新、加功能、改文案都不需要。**

### 三條鐵則

**① `versionCode` 一定要 +1**
版本代碼用過就不能重複。目前已用掉 `1`，下次填 `2`，只能往上加。

**② 一定要用同一把 `.keystore`**
PWABuilder 產生新套件時要上傳既有金鑰，不要讓它產生新的。換金鑰的後果：Play Console 拒收更新，且 `.well-known/assetlinks.json` 的 SHA-256 指紋對不上，TWA 網域驗證失效，App 上方會出現瀏覽器網址列。

金鑰檔已在 `.gitignore` 排除（`*.keystore` / `*.jks` / `*.pepk`），**絕不可進版控**。

**③ 套件名稱永遠是 `tw.searchbefore.app`**
改了就是另一個 App，無法挽回。

### 網域驗證相關檔案

| 檔案 | 用途 |
|---|---|
| `.well-known/assetlinks.json` | 證明 App 與 `searchbefore.tw` 屬於同一擁有者 |
| `.nojekyll` | **不可刪除**。沒有它，GitHub Pages 會忽略 `.` 開頭的資料夾，`.well-known` 會靜默 404，驗證直接失敗且無錯誤訊息 |

驗證是否仍有效：

```bash
curl -I https://searchbefore.tw/.well-known/assetlinks.json
```

`tests/index-syntax.test.js` 已加入 `.nojekyll` 存在性與 `assetlinks.json` 格式檢查。

## 五、Play Console 相關

| 項目 | 值／位置 |
|---|---|
| 套件名稱 | `tw.searchbefore.app`（**不可更改**） |
| 收費模式 | 免費（**發布後不可改為付費**；內購不受此限） |
| 應用程式類別 | 工具（Tools）。**不可選醫療／健康與健身／財經**，那些類別只能由機構帳戶發布 |
| 商店文案與素材 | `D:\Users\xiao6\Downloads\噴前查\play-assets\` |

### 更新商店文案或截圖

不用重新上傳 .aab。Play Console →「商店設定 →主要商店資訊」直接改，存檔後送審，通常一兩天生效。

### 截圖重新產生

素材是用無頭 Chrome 對真實 `index.html` 渲染的，畫面改版後要重跑。腳本邏輯：複製 `index.html`、注入 CSS/JS 設定畫面狀態、Chrome `--headless` 截圖，**不修改原始檔**。

| 用途 | 尺寸 |
|---|---|
| 手機 | 1080×1920（CSS 540×960 @2x） |
| 7 吋平板 | 1080×1920（CSS 720×1280 @1.5x） |
| 10 吋平板 | 1440×2560（CSS 720×1280 @2x） |
| 功能圖片 | 1024×500 |

比例必須是 16:9 或 9:16。

## 六、快速對照

**只改了資料或功能** → 改檔 → 三處版本號 → `node tests/run-all.js` → PR → 合併 → 手機驗證。結束。

**要改 App 名稱／圖示，或 Google 通知 target SDK 到期** → 上面全做完，再回 PWABuilder 用**原金鑰**產生 `versionCode +1` 的 .aab → Play Console 建立新版本 → 送審。
