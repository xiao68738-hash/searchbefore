# 資料與版本更新流程

更新日期：2026-07-21

噴前查有**三層**，各層的更新方式完全不同。搞錯層級是最常見的浪費時間來源。

| 層級 | 內容 | 更新方式 | 使用者何時看到 |
|---|---|---|---|
| ① 資料 | 農藥登記清單（`index.html` 內的 `DATA`） | 改 `index.html` → PR → 合併 | 下次開啟 App |
| ② 網站程式 | 畫面、功能、文案、各 `.js` | 改檔 → PR → 合併 | 下次開啟 App |
| ③ Android 外殼 | TWA 套件（.aab） | 重新產生 → Play Console 送審 | 使用者更新 App |

**重點：①②不用經過 Google 審查。** TWA 只是一層殼，App 打開時載入的是 `searchbefore.tw` 的即時內容，所以資料更新、加功能、修文案全部走 GitHub，合併完就生效。③ 幾乎用不到，見最後一節。

---

## 一、農藥資料更新（最常做的）

### 資料在哪裡

| 項目 | 位置 |
|---|---|
| 資料本體 | `index.html` 第 985 行 `const DATA={...}`（單行，約 3.6 MB） |
| 資料版本標記 | `index.html` 第 990 行 `const DATA_VERSION="2026-07"` |

### `DATA` 的結構

```
DATA[作物名][病蟲害名] = [ { name, form, content, dilution, phi, dose, times, moa, note, bl }, ... ]
```

規模：**284 種作物 / 2,801 組作物×病蟲害 / 16,342 筆藥劑**。
`bl` 是商品名陣列，`phi` 是安全採收期天數（可為 `null`），`moa` 是 IRAC／FRAC 代碼。

### ⚠️⚠️ 最大的缺口：`DATA` 的來源不明

**repo 裡沒有任何檔案或腳本能產生 `DATA`，也沒有記錄它是從哪個資料集轉來的。**

已確認 `mrl-data/` 裡的快照**都不是** `DATA` 的來源：

| 檔案 | 實際內容 | 有 作物/病蟲害/稀釋倍數/安全採收期 嗎 |
|---|---|---|
| `pesticides-latest.json` | 農業部 dataset 7293「農藥資料查詢」＝**許可證**清單（`permitNumber` / `form` / `content` / 廢止狀態） | ❌ 完全沒有 |
| `latest.json` | 食藥署 dataset 8944 殘留容許量 | ❌ |
| `crop-categories-latest.json` | 食藥署 dataset 8940 農產品分類表 | ❌ |
| `mrl-exempt-latest.json` | 食藥署 dataset 8943 免訂容許量 | ❌ |

`DATA` 需要的「作物 → 病蟲害 → 稀釋倍數 / 每公頃用量 / 安全採收期」屬於**農藥使用範圍及方法**這類資料集，目前沒有被抓取，也沒有留下轉換紀錄。

**這代表下次要更新農藥資料時，得先重新找出正確的來源資料集。** 這是整個專案最高的維護風險，優先於任何新功能。

建議的補救順序：

1. 確認正確的來源資料集（農藥使用範圍及方法），把端點記在這份文件裡
2. 寫 `scripts/build-data.mjs`：抓取 → 正規化成上面的結構 → 寫回 `index.html` 第 985 行
3. 寫一支比對腳本，報出新舊 `DATA` 的差異筆數，更新前先看差異合不合理

### 另一個未解決的問題：9,999 筆上限

`pesticides-latest.json` 目前 `count: 9999`，且自帶警告：

> 來源剛好回傳 9,999 筆，可能是舊版端點上限；任何查無結果都只能列為尚未解析。

已確認 `mrl-data/*.mjs` 裡**沒有任何分頁處理**（沒有 `$skip`、沒有分頁迴圈）。也就是說這份快照很可能是**不完整**的，MRL 對照裡「查無資料」的項目有一部分可能只是被截斷了。

重新抓取時要先確認端點的分頁參數，並驗證總筆數不等於 9,999。

### 更新步驟（來源確定之後）

1. 從 `main` 開分支，例如 `data/2026-10`
2. 產生新的 `DATA`，替換 `index.html` 第 985 行整行
3. 把 `DATA_VERSION` 改成抓取年月，例如 `"2026-10"`
4. 執行下面「三處版本號」與「測試與發布」

## 二、三處版本號（每次更新都要）

漏改任何一處，使用者手機上會吃到舊快取，你在電腦看到更新了但農友沒有。

| 檔案 | 位置 | 規則 |
|---|---|---|
| `index.html` | `const APP_VERSION="…"` | 往上加 |
| `sw.js` | `const CACHE_VERSION = "…"` | 換成新字串，習慣帶用途與日期 |
| `tests/index-syntax.test.js` | **兩處**釘住值 | 必須同步改，否則測試失敗 |

**測試檔裡有兩個釘住值，很容易只改到一個：**

| 行號 | 內容 | 釘住的是 |
|---|---|---|
| 第 31 行 | `assert.match(html, /const APP_VERSION="…"/)` | `index.html` 的 `APP_VERSION` |
| 第 112 行 | `assert.match(sw, /v0\.1\.9\.1-crop-alias/)` | `sw.js` 的 `CACHE_VERSION` |

第 112 行**沒有出現 `CACHE_VERSION` 這個字**，只寫版本字串本身，所以用關鍵字搜尋 `CACHE_VERSION` 是找不到它的。要搜就搜版本號本身。

好消息是兩者都有測試保護，忘了改一定會被 `node tests/run-all.js` 擋下來，不會靜默漏掉。

## 三、測試與發布

```bash
node tests/run-all.js          # 全部測試，必過才能合併
npm run release:check          # 測試 + 建置 + 發布成品檢查
```

`release:check` 會額外驗證：dist 不含文件或 source map、沒有私鑰外洩、沒有殘留舊網址、`index.html` 引用的 script 與 Service Worker 快取清單都存在。

### GitHub 流程（**永遠不要直接改 `main`**）

1. 從 `main` 開工作分支
2. 改動 + 測試通過
3. Commit，建立 Pull Request
4. 確認差異後合併到 `main`
5. GitHub Pages 自動部署，網站即更新
6. 確認網站正常後刪除分支

### 部署後自我檢查

- 手機開啟網站，確認新資料／新功能出現
- 關網路重開，確認離線仍可用（Service Worker 有正確更新）
- 從 Android App 開啟，確認**上方沒有跑出瀏覽器網址列**（有的話代表 TWA 網域驗證出問題，見下節）

## 四、Android 外殼更新（很少做）

### 什麼時候才需要重新產生 .aab

只有這幾種：

- App 名稱或圖示改了
- Google 每年強制拉高的 **target SDK 門檻**（約每年 8 月底，Play Console 會提前寄信，目前是 35）
- TWA 設定變動（啟動網址、螢幕方向、狀態列顏色）
- 未來要做內購

**資料更新、加功能、改文案都不需要。**

### 三條鐵則

**① `versionCode` 一定要 +1**
版本代碼用過就不能重複。目前已用掉 `1`，下次填 `2`，只能往上加。

**② 一定要用同一把 `.keystore`**
PWABuilder 產生新套件時要上傳既有金鑰，不要讓它產生新的。換金鑰的後果：Play Console 拒收更新，且 `.well-known/assetlinks.json` 的 SHA-256 指紋對不上，TWA 網域驗證失效，App 上方會出現瀏覽器網址列。

金鑰檔已在 `.gitignore` 排除（`*.keystore` / `*.jks` / `*.pepk`），**絕不可進版控**。

**③ 套件名稱永遠是 `tw.searchbefore.app`**
改了就是另一個 App，無法挽回。

### 網域驗證相關檔案

| 檔案 | 用途 |
|---|---|
| `.well-known/assetlinks.json` | 證明 App 與 `searchbefore.tw` 屬於同一擁有者 |
| `.nojekyll` | **不可刪除**。沒有它，GitHub Pages 會忽略 `.` 開頭的資料夾，`.well-known` 會靜默 404，驗證直接失敗且無錯誤訊息 |

驗證是否仍有效：

```bash
curl -I https://searchbefore.tw/.well-known/assetlinks.json
```

`tests/index-syntax.test.js` 已加入 `.nojekyll` 存在性與 `assetlinks.json` 格式檢查。

## 五、Play Console 相關

| 項目 | 值／位置 |
|---|---|
| 套件名稱 | `tw.searchbefore.app`（**不可更改**） |
| 收費模式 | 免費（**發布後不可改為付費**；內購不受此限） |
| 應用程式類別 | 工具（Tools）。**不可選醫療／健康與健身／財經**，那些類別只能由機構帳戶發布 |
| 商店文案與素材 | `D:\Users\xiao6\Downloads\噴前查\play-assets\` |

### 更新商店文案或截圖

不用重新上傳 .aab。Play Console →「商店設定 →主要商店資訊」直接改，存檔後送審，通常一兩天生效。

### 截圖重新產生

素材是用無頭 Chrome 對真實 `index.html` 渲染的，畫面改版後要重跑。腳本邏輯：複製 `index.html`、注入 CSS/JS 設定畫面狀態、Chrome `--headless` 截圖，**不修改原始檔**。

| 用途 | 尺寸 |
|---|---|
| 手機 | 1080×1920（CSS 540×960 @2x） |
| 7 吋平板 | 1080×1920（CSS 720×1280 @1.5x） |
| 10 吋平板 | 1440×2560（CSS 720×1280 @2x） |
| 功能圖片 | 1024×500 |

比例必須是 16:9 或 9:16。

## 六、快速對照

**只改了資料或功能** → 改檔 → 三處版本號 → `node tests/run-all.js` → PR → 合併 → 手機驗證。結束。

**要改 App 名稱／圖示，或 Google 通知 target SDK 到期** → 上面全做完，再回 PWABuilder 用**原金鑰**產生 `versionCode +1` 的 .aab → Play Console 建立新版本 → 送審。
