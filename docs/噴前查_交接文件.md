更新日期：2026-07-21
目前功能版本：`v1.5.0`（現有功能串接）

## 專案概況

噴前查是給台灣農友使用的 PWA，功能包括農藥登記查詢、稀釋計算、施藥紀錄、安全採收期倒數，以及栽培、施肥、採收、採後處理、資材購入等田間作業紀錄。

- GitHub：https://github.com/xiao68738-hash/searchbefore
- 正式網站：https://searchbefore.tw/
- 農藥資料來源：農業部農藥開放資料

## 不可破壞的安全規則

1. 粉劑以公克 `g` 顯示，絕不可換算成瓶蓋、茶匙或湯匙。
2. 只有液劑 `ml` 可以顯示容量估算，實際配製仍以量具與標示為準。
3. 採收備註若列出露天、設施或不同用途天數，在沒有栽培方式選項前採最長天數。
4. 同一田區或種植批次的每一筆施藥都必須納入計算，取最晚可採日期。
5. 任一筆施藥紀錄缺少採收期時，該批次不得顯示「可採收」。
6. `track`、`notify` 或畫面篩選只能控制提醒與顯示，不能把實際施藥排除於安全計算。
7. 作物群組 `PARENT` 尚未逐項完成官方校驗，目前不可自動合併上層群組登記。
8. 採收事件即使觸發安全警告仍可留存，但不得省略警告與判定快照。
9. 收穫型態(如蒜頭/蒜葉、乾豆/毛豆)只依備註「明確文字」分類或排除;備註未註明者不得自動歸入任何型態(見 `docs/作物收穫型態消歧.md`)。
10. 型態分組只影響顯示順序與提醒,不得移除任何藥劑;「不符/待確認」須可展開查看並附排除理由,且分組不得改變藥劑索引(帶入計算/紀錄須對到同一筆)。

## GitHub 保存與發布流程

1. `main` 永遠保留可公開使用的穩定版。
2. 從 `main` 建立一個工作分支，例如 `feature/connected-records`。
3. 在工作分支完成修改與測試。
4. 保存成有意義的 Commit。
5. 建立草稿 Pull Request，檢查差異及測試結果。
6. 確認後合併到 `main`，GitHub Pages 才會更新正式網站。
7. 網站確認正常後，刪除已完成的工作分支並建立版本標籤。

| 儲存鍵 | 內容 | 是否同步到雲端 |
|---|---|---|
| `fieldPlots` | 田區、作物、種植日期與批次 | ✅ |
| `records` | 每次實際施藥；同批次全部紀錄共同決定最晚可採日 | ✅ |
| `farmRecords` | 栽培、施肥、採收、採後、資材購入；採收可帶 `safetyCheck` 快照 | ✅ |

| `syncTombstones` | 刪除標記，供同步傳播用 | 內部 |

### v4 新增的兩個欄位

**`updatedAt`** — 每筆紀錄的最後修改時間。雲端同步靠它判斷新舊。

**刪除標記** — 刪除的紀錄會在 `syncTombstones` 留下記號。沒有這個，A 手機刪掉的紀錄會被 B 手機同步回來。

兩者都由包裝過的 `store.set` 自動維護，**不需要在各處 mutation 路徑手動處理**。新增功能時照常呼叫 `store.set` 即可。

## 雲端同步

登入後預設開啟，個人頁可關閉。Firestore 結構 `users/{uid}/{collection}/{docId}`。

### 三個不可違反的設計

**① 本機優先。** `localStorage` 永遠是可直接使用的完整資料，沒有網路照常運作。

**② 時戳必須嚴格遞增。** `Date` 只有毫秒解析度，同一毫秒的連續寫入會拿到相同值；手機時鐘往回調時，新編輯會輸給舊資料而**靜默回滾使用者的輸入**。`cloud-sync.js` 的 `nowIso()` 已處理。

**③ 時戳相同時刪除優先。** 否則 A 裝置刪掉的紀錄會被 B 裝置同步回來，變成刪不掉。

### 免費額度

Spark 方案為讀 50,000／寫 20,000 次一天。同步採**增量讀取**（只查 `updatedAt >` 上次同步時間）且**一致資料不重傳**。改動 `mergeCollection` 前先看 `tests/cloud-sync.test.js` 裡關於額度的那幾條測試——放寬條件會直接讓額度爆掉。
