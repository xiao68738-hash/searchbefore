## 一、這個功能要解決什麼

App 現在的農藥登記資料回答「某藥是否核准用在某作物與害物」；MRL 資料回答「某有效成分在某農產品的殘留容許量」。

兩者不能只靠畫面上的中文藥名直接合併。正確資料鏈必須是：

```text
App 農藥普通名稱
→ 官方農藥許可證
→ 一種或多種英文有效成分
→ MRL／免訂容許量清單
→ 特定作物／官方作物分類／通用其他類別
→ 仍有歧義就回傳「無法確認」
```

## 三、先前直接比對方法為什麼不夠

### 1. 混合劑不是一個 MRL 有效成分

App 中的 `三氟得克利`、`白列克敏` 等名稱可能代表混合劑。MRL 則依個別有效成分訂定。

因此必須先用官方許可證資料把中文名稱拆成英文有效成分，不能靠中文字拆字或猜測。

### 2. 中文 MRL 欄位有缺字

目前快照有大量 `?` 缺字，中文萬用字元比對只能列為人工候選。

MRL 的 `國際普通名稱` 相對完整，因此內部主要對照鍵採正規化後的英文名稱；中文名稱只用於顯示與稽核。

### 3. 沒有特定作物同名列，不等於立刻判定「不得檢出」

MRL 另有官方作物分類及 `其他蔬果類`、`其他穀類` 等通用列。帶 `*` 的數值還涉及公告檢驗方法定量極限。

判斷順序至少必須包含：

1. 特定作物完全相符。
2. 官方作物分類相符。
3. 適用的「其他⋯類」通用列。
4. 免訂容許量清單。
5. 以上均無法可靠確認時，回傳「無法確認」並交人工稽核。

階段 0 不產生「不得檢出」的自動結論。

### 4. 官方農藥舊端點可能有筆數上限

目前舊版下載端點剛好回傳 9,999 筆，可能是來源上限。因此即使名稱查無，也只能標為「尚未解析」，不能推論該農藥不存在或沒有 MRL。

## 四、階段 0 已完成的工作

- MRL 快照下載、欄位逐筆驗證、逾時及重試。
- 農藥許可證、作物分類與免訂容許量三份參考快照。
- 快照時間與 SHA-256 內容雜湊。
- 同日重跑仍保留到秒的獨立快照。
- 中文農藥名稱 → 官方英文有效成分組合。
- 混合劑以 `+` 拆成個別成分。
- 同名多組成分、名稱查無、英文成分未對到、作物尚未分類等完整稽核。
- 人工清單同時輸出 Markdown、CSV、JSON，不再只列前 40 筆。
- MRL 基礎對照單元測試。

目前這些資料仍由 `.firebaseignore` 排除，不會進入正式網站。

## 六、下一階段

### 階段 1：先完成可靠的作物分類器

- 將 App 作物精確對到官方分類表。
- 為 `其他⋯類（⋯除外）` 建立可測試的優先順序。
- 先處理蒜／蒜頭等已完成收穫型態消歧的案例。
- 每個分類結果必須能說明命中哪一條官方資料。

### 階段 2：建立只讀 MRL 查詢結果

- 只在資料鏈全程明確時顯示 MRL 原文與來源。
- 顯示有效成分、作物匹配方式、ppm 原字串、資料快照時間。
- 帶 `*` 的容許量保留原字串與註解，不做一般數字比較。
- 未解析一律顯示中性「無法確認」。

### 階段 3：小規模人工驗證

- 選少量作物逐筆比對官方原文。
- 與植物保護／農藥殘留專業人員覆核。
- 驗證完成前不宣稱可代替法規查詢或檢驗判定。

## 七、不可違反的安全規則

1. 不從中文混合劑名稱猜測有效成分。
2. 不把「名稱查無」直接改寫成「未訂容許量」或「不得檢出」。
3. 不忽略官方作物分類、通用其他類別及免訂清單。
4. 不把安全採收期與 MRL 混為同一判斷。
5. 資料未載入、版本不明或任一層對照失敗時，不顯示綠色安全結論。
6. 所有結果都必須保留來源資料集、快照時間與對照路徑。
7. MRL 只描述標準值，不代表依標示施藥後一定能通過實際檢驗。
